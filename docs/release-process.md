# 发布流程

本仓库以后按这个流程发版。不要为了一个很小的页面文字改动连续发很多 Release。

## 什么时候发 Release

可以发：

- 安装脚本变化。
- 前端覆盖脚本变化。
- 后端补丁变化。
- 用户能明显感知到的功能变化。
- 安全修复。
- 发布流程、CI、隐私规则这类会影响维护方式的变化。

不要单独发：

- README 小字修改。
- 错别字。
- 临时说明。
- 不影响安装和使用的轻微文档调整。

这些小改动先进入 `CHANGELOG.md` 的 `Unreleased`，等下一次正式版本一起发。

## 版本号规则

- `v主版本.次版本.修订版本`
- 修复小问题：增加修订版本，例如 `v1.1.1`
- 增加功能或发布流程变化：增加次版本，例如 `v1.2.0`
- 破坏兼容或安装方式大改：增加主版本，例如 `v2.0.0`

## 发布前必须检查

发布前必须完成：

1. 确认仓库是公开还是私有，公开仓库不能带任何私密信息。
2. `git status` 必须只包含本次相关改动。
3. 运行语法检查：
   - `bash -n install.sh`
   - `node --check metube-zh-cn.js`
   - `python -m py_compile ytdl.py subscriptions.py dl_formats.py`
4. 扫描敏感信息：
   - 服务器地址
   - 密码
   - Cookie
   - API Key
   - SSH Key
   - 安装日记
5. 更新 `CHANGELOG.md`。
6. 如果安装脚本或前端缓存版本变化，确认版本号已经同步。
7. Release Notes 必须写清：
   - 这次改了什么
   - 用户是否需要重装或刷新
   - 验证结果
   - 回滚办法

## 正式发布步骤

1. 本地检查通过。
2. 提交代码。
3. 推送 `main`。
4. 等 GitHub Actions 质量门禁通过。
5. 创建 Release，Release Notes 从 `docs/releases/版本号.md` 读取。
6. 发布后确认：
   - Release 不是 Draft
   - Release 不是 Prerelease
   - 最新 tag 指向正确提交
   - 仓库里搜不到敏感信息

## 回滚规则

如果服务器功能出问题：

1. 不直接删文件。
2. 用 `/www/临时文件` 里的备份恢复明确文件。
3. 恢复后重新检查服务状态。
4. 再补一个修复提交，写清原因。

如果仓库发布说明写错：

1. 优先编辑 Release Notes。
2. 不删除历史 Release，除非里面有敏感信息。
3. 如果真的泄露敏感信息，先撤掉公开内容，再轮换相关密钥。
